# patterns

Named architectural patterns this repository is built on. Each concept
explains one load-bearing idea — what it is, why it's shaped that way, and
where it's implemented.

## Concepts

* [Cross-OS Module Twins](cross-os-module-twins.md) - A cross-platform feature is a twin pair — one module per class dir (modules/darwin/<name>.nix ↔ modules/nixos/<name>.nix) — with non-trivial shared text extracted to pure builders in lib/ and package lists kept in sync by hand.
* [Dendritic Module Layout](dendritic-modules.md) - Every .nix file under modules/ is auto-discovered as a flake-parts module via import-tree — no manual import lists anywhere.
* [Host-Mounted Modules](host-mounted-modules.md) - Universal features are ungated deferred modules in flake.modules.<class>.* (darwin + nixos classes); host-selective features gate behind programs./services. enable options flipped in modules/hosts/<hostname>/default.nix; truly host-specific files live beside the host's registration.
* [Host Registry Realisers](host-registry-realisers.md) - Hosts register into typed configurations.<class>.<hostname> option registries; per-class realiser modules (darwin.nix via darwinSystem, nixos.nix via snowglobe-lib's mkNixosHost) turn each entry into a flake output plus a per-host flake check.
* [Snapshot-Synced Configs](snapshot-synced-configs.md) - Apps that rewrite their config via atomic rename (Helium, Noctalia) can't be stowed — settings are git-tracked as snapshots under config/, synced explicitly with per-app capture/restore/diff CLIs; Helium's snapshots are age-encrypted at rest.
* [Store-path-embedding Configs](store-path-configs.md) - Config files that must reference /nix/store paths are generated by the feature's per-class module and linked into place — darwin during activation (postActivation, order 1600), nixos via systemd.tmpfiles L+ rules — not stowed.
* [Stow Tree](stow-tree.md) - Plain config files live in home/ as GNU Stow packages symlinked into $HOME — one tree shared by macOS and NixOS, pointing at the live repo so edits apply without a rebuild.
* [Sub-flake Extraction](subflake-extraction.md) - Packages destined for standalone life live under flakes/<name>/ as self-contained flakes consumed via relative-path inputs — extraction to a separate repo is a one-line URL swap.
