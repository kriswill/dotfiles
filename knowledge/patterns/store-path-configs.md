The [stow tree](stow-tree.md) links plain files from the live repo — but a
file that must embed a `/nix/store` path (a plugin path, a wrapped binary, a
rendered option value) would go stale the moment the store path changes.
Those files are instead **generated by the owning feature module**
(`pkgs.writeText` or rendered from module options) and symlinked into place
during activation, so every rebuild refreshes them atomically.

The split in practice:

- [tmux](../modules/tmux.md): static config is stowed; `plugins.conf` — which
  references plugin store paths — is generated and linked next to it.
- [claude-account-selector](../modules/claude-account-selector.md): the zsh
  wrapper is a plain file, but its host-specific rule preamble is rendered
  from Nix options and prepended via `writeText`.
- [podman-desktop](../modules/podman-desktop.md): GUI settings that reference
  packaged binaries are generated; the rest is stowed.

Rule of thumb: **if the file's content depends on evaluation output, the
module owns it; otherwise the stow tree owns it.** Each generated file lives
beside its stowed siblings so the app sees one coherent config directory.
